# Contour Graph — концепция

## ida-procon: что работает

ida-procon решает задачу автономного реверс-инжиниринга бинарников. IDA Pro декомпилирует код, но на выходе тысячи безымянных функций без контекста. ida-procon дампит всё один раз, закрывает IDA, и дальше LLM-агенты работают параллельно через координатор.

Координатор — FastAPI демон с пятью сервисами в памяти. Registry загружает манифест (граф всех функций с callees и xrefs) и coverage (какие функции обработаны, какие нет). Claimer даёт атомарные захваты с TTL: агент claim'ит функцию, у него 600 секунд, потом блокировка снимается автоматически — защита от зависших агентов. WriteQueue сериализует все мутации в один поток и батчит запись на диск. Resolver обновляет статусы функций. ContourBuilder собирает контуры — именованные подграфы связанных функций, представляющие одну логическую единицу бинарника.

Ключевой эндпоинт — /next-entry. Он сортирует непокрытые функции по "богатству": сколько у функции непокрытых callees. Агент всегда получает самую ценную точку входа. Агент не выбирает что анализировать — координатор решает за него на основе метрик графа.

Агент работает по циклу: запросил entry → claim → трассировал callees → улучшил код → submit contour. Контур содержит ноды с ролями (entry, helper, leaf, micro, borrowed), рёбра, external_deps, summary. Micro — функция на десять строк, которая шарится между контурами, но не становится отдельной единицей. Borrowed — функция, захваченная другим агентом, доступна только для чтения.

Результат на реальном бинарнике: 6 агентов, 15 минут, 180 контуров, 948 функций задокументировано, $6 в токенах. Агенты сами распознали FFT, SVD, Cholesky, нейросети, Qt/COM мосты — без подсказок.

Главный вывод из ida-procon: агент эффективен когда получает узкий контекст (одна функция + её callees), а не весь проект. Координатор управляет не агентами, а их полем зрения.

Второй вывод, практический: claim в ida-procon — блокирующий, потому что два агента не должны одновременно перезаписывать одну функцию в файле. Это решение конкретной проблемы write-конфликта в файловой системе. Когда код живёт в БД и агент не перезаписывает ноду, а создаёт новую версию — блокировка не нужна. Десять агентов могут одновременно работать над одной нодой, каждый в своей версии. Claim превращается из блокировки в маркер загрузки: координатор знает кто над чем работает, но не запрещает параллельную работу.

Третий вывод, о контроле: когда 18 агентов потеряли связь с координатором, часть из них обошла API и начала писать напрямую в файлы. Некоторые рестартили убитый сервер, меняли ему порт, искали обходные пути. Файловая система — это fallback, который агенты находят и используют. Когда единственный путь к коду — API, а файлов нет, этот fallback исчезает. Изоляция кода в БД — это не только архитектурное решение, это контроль над агентами.

## MCGK: что работает

MCGK — прокси-сервер, единственный хаб связи между независимыми сервисами. Ни один сервис не знает где живёт другой — ни адреса, ни порта. Сервис знает только адрес MCGK и имя того, с кем хочет поговорить.

При старте сервис регистрируется в MCGK через POST /register, отправляя паспорт: имя, адрес (только для MCGK, никогда не отдаётся наружу), описание, список эндпоинтов с описаниями и JSON-schema контрактов (accepts/returns). MCGK хранит паспорта в SQLite (WAL mode), при рестарте восстанавливает реестр.

Другой сервис может узнать паспорт через GET /discover/{name} — получит описание, эндпоинты, схемы, но никогда адрес. Запрос отправляется через /route/{name}/{path} — MCGK проксирует, сохраняя заголовки, query string, тело, метод, стриминг. GET /map — все сервисы и их статусы. GET /observe — логи всех запросов. GET /health — результаты health-чеков.

Философия melting contract: вызывающий адаптируется к вызываемому, не наоборот. Вызывающий получает паспорт, читает схему, формирует запрос под неё. Если ответ не в ожидаемом формате — вызывающий адаптирует свой парсер. MCGK не трансформирует данные, не валидирует тела, не кэширует. Он только маршрутизирует и хранит паспорта.

Главный вывод из MCGK: полная изоляция адресов + паспорта с контрактами = система, в которой сервисы можно добавлять, убирать и заменять без каскадных изменений. Каждый сервис автономен, связь только через имена.

## Код без файлов: что мы поняли

Файловая система — это интерфейс для людей. Человек ориентируется по директориям, именам файлов, пространственной памяти ("auth где-то в src/utils"). LLM-агент не имеет пространственной памяти. Когда агент получает проект — он видит плоский поток текста. Файловая структура для него шум, а навигация по ней жрёт контекст, который мог бы тратиться на решения.

Когда агент работает с файлом на 500 строк, три четверти контекста уходит на то чтобы найти нужное место. Импорты занимают десятки строк и не несут логической информации — это метаданные сборки, а не метаданные смысла. Агент их читает, тратит токены, и не получает ничего полезного для решения задачи.

ida-procon уже решила эту проблему для мёртвого кода: агент не открывает файл, он делает GET /func-code и получает одну функцию. Меняет — POST /update-code. Метаданные хранятся отдельно в манифесте. Координатор собирает контур через /contour-code, склеивая функции в правильном порядке.

То же самое работает для живого кода. Код хранится в БД как граф нод. Нода — минимальная единица кода, содержащая хотя бы одно решение (ветвление, обработка ошибки, итерация с условием). Если код не содержит решения (чистая арифметика, форматирование, маппинг) — это micro, часть другой ноды, не самостоятельная единица. Давление сверху на размер ноды — тест (если нода не тестируема изолированно — слишком мелкая). Давление снизу — decision point (если нет ни одного решения — это не нода).

Имя ноды — глагольное, описывающее действие: validate_jwt, calc_r2, render_table. Только what_do, без домена и размера в имени. Тематическая принадлежность выражается тегами: validate_jwt может быть одновременно auth, api и security. Теги — контролируемый словарь с давлением на реиспользование: при создании тега сервис ищет существующие по fuzzy match и предлагает использовать имеющийся. Новый тег — только если ни один не подходит. Периодически агент пробегает по всему словарю и нормализует дупликаты. Цель — держать количество тегов в управляемых пределах, чтобы поиск похожих нод не деградировал с ростом графа.

Каждая нода хранит чистый код без импортов, контракт (что принимает, что возвращает), список зависимостей (какие другие ноды вызывает), теги и спецификации четырёх уровней гранулярности. Уровни вдохновлены числами Данбара — вложенная структура социальных связей (5/15/50/150) где каждый уровень даёт меньше деталей. Для нод: as_is (полный код + все спеки), summary (полное описание без кода), ticket (15 строк — что делает, контракт, edge cases), name (имя ноды — само по себе достаточно говорящее, чтобы передать суть без дополнительного текста). Граф из имён нод и рёбер между ними — уже читаемая карта: кто кого вызывает и зачем, без единого байта описания.

Код и спеки живут в одном графе, в одном сервисе. Разные уровни детализации — это не разные хранилища, а разные эндпоинты одного API. Агент не может случайно загрузить весь граф в as_is — он получает ровно тот срез, который запросил вызывающий. Контроль плотности контекста — задача API, не архитектуры хранения.

Импорты генерируются автоматически при сборке. Скрипт обходит граф, резолвит зависимости, собирает файлы в формате целевого фреймворка, запускает бандлер. Файлы — промежуточный артефакт сборки, не источник правды. Как .o файлы при компиляции C.

Версионирование на уровне нод, не файлов. Агент меняет ноду — создаётся новая версия с task_id. Версия проходит жизненный цикл: draft → golden (принята, используется для сборки и контекста) → deprecated (заменена новой golden). Или draft → rejected (не прошла тесты, хранится как failure-indexed memory, удаляется через 7 дней). Несколько агентов параллельно пишут разные версии одной ноды — турнир: каждая версия прогоняется через тесты с golden-окружением, лучшая побеждает. Не мерж текста, а отбор по поведению. Таблица nodes хранит кэш активной версии (golden, или draft если golden нет) для быстрого чтения. Таблица versions — source of truth и аудит-лог. Синхронизация при promote/reject.

Один граф — много продуктов. Продукт — это подмножество нод плюс свои точки входа. Контур auth_flow используется в трёх продуктах. Добавить новый продукт — выбрать ноды, добавить недостающие, определить рёбра.

Проекция для человека генерируется по запросу: карта контуров, описания на человеческом языке, полный проект в файлах для скачивания. Человек не обязан читать код — он видит что система делает и почему, через спеки, не через исходники.

## Данбар подтверждён экспериментально

Серия экспериментов (06, 07, 08) показала что Данбар-контекст — не компромисс, а оптимум. Результат — inverted U curve: полный код проекта (2000 строк) даёт оценку 18/25, обрезанный код (300 строк) — 14/25, Данбар-пакет (summary + contracts) — 23/25, только контракты — 14/25.

Механизм: полный код провоцирует копирование существующих паттернов. Агент видит main() и воспроизводит его подход, даже если подход плохой. Данбар-контекст заставляет агента думать архитектурно — он знает что система делает, но не видит как, и проектирует решение заново. Это регулярно даёт более качественный результат чем копирование.

Критически важно что лежит на depth 0 (целевая нода). Код на depth 0 якорит понимание агента. Метаданные на depth 0 (JSON-шаблон класса) — слабее. Quantity на depth 1 вторична по отношению к quality на depth 0.

## OOP-декомпозиция: гибрид A+B

Классы, как и файлы, являются проекцией для человеческой навигации. В рантайме ООП компилируется в плоские вызовы функций. Граф не нуждается в классах как сущности — ему нужны функции с явными контрактами.

Декомпозиция ООП-кода использует подход A (Maximum Decomposition) как основу: каждый метод становится standalone-функцией, self заменяется на явный аргумент state, dispatch и наследование выражаются через рёбра и dispatch-ноды. Это даёт стабильный "пол" качества — агенты получают однозначный контекст и пишут предсказуемый код.

Поверх структуры A — обогащённые спеки в стиле подхода B (Contract-Boundary): на граничных нодах (dispatch-точки, мосты между паттернами, event publisher/subscriber, entry-точки пайплайнов) spec_summary содержит архитектурную информацию — принадлежность к паттерну, sibling-ноды, мосты к другим паттернам, инварианты. Это ~25% нод графа. Остальные 75% обходятся обычными спеками.

Обогащённые спеки дают агенту шанс на B2-уровень — архитектурно глубокий код, интегрированный с паттернами системы. Турнир версий делает это безрисковым: координатор запускает двух солдат (один с обычным контекстом, один с обогащённым), оба проходят тесты, выбирается лучший. Пол A гарантирован, потолок B не ограничен.

Обогащение спеков — задача капрала при декомпозиции и при ревью после работы солдат. Периодический sweep обновляет устаревшие спеки (паттерны меняются, мосты перестраиваются). Это реальная стоимость подхода.

Для фреймворк-кода (PyQt, Django) где класс обязан существовать из-за требований фреймворка: бизнес-логика извлекается в функции-ноды, а тонкий класс-адаптер содержит только делегирование. Адаптер — это не нода, а build-time проекция, собираемая builder'ом из функций-нод и метаданных реконструкции.

## Композиция: Contour Graph

Contour Graph — система, в которой MCGK является единым гейтом, а за ним стоят сервисы, каждый со своей ответственностью. Все сервисы регистрируются через паспорта, общаются только по именам, адреса скрыты.

MCGK Gate — уже готов. Форк существующего MCGK. Без изменений.

CodeGraph — единый сервис хранения кода, спецификаций и версий. Хранит ноды графа: id, код, язык, kind (contour/micro/config), контракт (accepts/returns), зависимости (imports), теги. Хранит рёбра между нодами (calls/uses/extends/tests). Хранит спецификации четырёх уровней. Хранит версии нод с жизненным циклом draft/golden/rejected/deprecated. Таблица nodes — кэш активной версии для быстрого чтения. Таблица versions — source of truth. Предоставляет эндпоинты: CRUD нод, управление версиями (POST /node/{id}/version, /promote, /reject, GET /versions), контекстный пакет по Данбару (GET /context/{id}), поиск, граф. Регистрируется в MCGK как "codegraph".

Builder — получает граф из codegraph, для каждой ноды берёт golden-версию, генерирует импорты, собирает файлы, запускает бандлер. Три режима: build (весь продукт из golden), test_build (одна нода заменена на draft-версию, остальное golden), build_contour (standalone фрагмент для быстрой проверки). Регистрируется в MCGK как "builder".

Tester — получает собранный артефакт от builder, прогоняет тесты. Тесты — тоже ноды в графе с edge_type "tests". Регистрируется в MCGK как "tester".

Coordinator — инфраструктурный сервис верхнего уровня. Назначает задачи (/next-task — нода с наибольшим количеством непокрытых соседей). Перед созданием новой ноды проверяет через codegraph /similar есть ли подходящая. Собирает контекстные пакеты через codegraph. Управляет жизненным циклом: draft → test_build → test → golden или reject. Управляет турнирами версий. Хранит маркеры загрузки — кто из агентов над чем работает — как информационные метки, не блокировки. Garbage collection: ноды без входящих рёбер и без golden-статуса удаляются по TTL, rejected-версии старше 7 дней чистятся. Регистрируется в MCGK как "coordinator".

Координатор не знает про ранги агентов. Он знает про ноды, рёбра и гранулярности. Ранги — логика агентов, не логика инфраструктуры.

## Ранги агентов

Агенты организованы в три ранга с убывающим контекстом и возрастающей конкретностью.

Sergeant — работает с сутью, никогда не пишет код. Общается с человеком, вытаскивает неявное знание, формулирует задачи. Имеет доступ к этому документу, к спекам, к карте графа — но не к коду. Запрашивает у координатора контекстные пакеты и формирует из них .md артефакты для капралов. Sergeant — пользователь инфраструктуры, не часть её.

Corporal — получает .md артефакт от сержанта или напрямую от человека. Знает конкретный спек, видит код, пишет код. Может вызывать субагентов ранга Soldier для параллельной работы внутри своей задачи. В 95% случаев это Sonnet; Opus — только когда задача требует широкого контекста, что случается редко.

Soldier — субагент капрала. Выполняет узкую конкретную подзадачу: написать одну ноду, прогнать тест, обновить спек. Не принимает архитектурных решений. Sonnet.

Данбаровские круги работают на двух уровнях: для нод (гранулярность спеков по дистанции в графе) и для агентов (объём контекста по рангу). Sergeant видит весь граф на уровне name и ticket. Corporal видит свой контур на уровне as_is и окрестности на уровне summary. Soldier видит одну ноду.

## Порядок поднятия

Сначала MCGK, потом codegraph, потом builder, потом tester, потом coordinator. Каждый следующий сервис зависит от предыдущих только через MCGK — по именам, не по адресам. Каждый можно поднять, протестировать и потрогать отдельно прежде чем двигаться дальше.

Текущее состояние: MCGK работает (форк, без изменений). CodeGraph работает (ноды, рёбра, версионирование, Данбар-контекст, поиск). Build.py — скрипт сборки из графа в один .py файл (не сервис). Builder, tester, coordinator — не реализованы. Данбар подтверждён экспериментально. Гибрид A+B выбран как подход к декомпозиции. Порядок следующих шагов определяется практикой.
