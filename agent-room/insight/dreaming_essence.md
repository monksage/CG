# Contour Graph — концепция

## ida-procon: что работает

ida-procon решает задачу автономного реверс-инжиниринга бинарников. IDA Pro декомпилирует код, но на выходе тысячи безымянных функций без контекста. ida-procon дампит всё один раз, закрывает IDA, и дальше LLM-агенты работают параллельно через координатор.

Координатор — FastAPI демон с пятью сервисами в памяти. Registry загружает манифест (граф всех функций с callees и xrefs) и coverage (какие функции обработаны, какие нет). Claimer даёт атомарные захваты с TTL: агент claim'ит функцию, у него 600 секунд, потом блокировка снимается автоматически — защита от зависших агентов. WriteQueue сериализует все мутации в один поток и батчит запись на диск. Resolver обновляет статусы функций. ContourBuilder собирает контуры — именованные подграфы связанных функций, представляющие одну логическую единицу бинарника.

Ключевой эндпоинт — /next-entry. Он сортирует непокрытые функции по "богатству": сколько у функции непокрытых callees. Агент всегда получает самую ценную точку входа. Агент не выбирает что анализировать — координатор решает за него на основе метрик графа.

Агент работает по циклу: запросил entry → claim → трассировал callees → улучшил код → submit contour. Контур содержит ноды с ролями (entry, helper, leaf, micro, borrowed), рёбра, external_deps, summary. Micro — функция на десять строк, которая шарится между контурами, но не становится отдельной единицей. Borrowed — функция, захваченная другим агентом, доступна только для чтения.

Результат на реальном бинарнике: 6 агентов, 15 минут, 180 контуров, 948 функций задокументировано, $6 в токенах. Агенты сами распознали FFT, SVD, Cholesky, нейросети, Qt/COM мосты — без подсказок.

Главный вывод из ida-procon: агент эффективен когда получает узкий контекст (одна функция + её callees), а не весь проект. Координатор управляет не агентами, а их полем зрения.

## MCGK: что работает

MCGK — прокси-сервер, единственный хаб связи между независимыми сервисами (контурами). Ни один сервис не знает где живёт другой — ни адреса, ни порта. Сервис знает только адрес MCGK и имя того, с кем хочет поговорить.

При старте сервис регистрируется в MCGK через POST /register, отправляя паспорт: имя, адрес (только для MCGK, никогда не отдаётся наружу), описание, список эндпоинтов с описаниями и JSON-schema контрактов (accepts/returns). MCGK хранит паспорта в SQLite (WAL mode), при рестарте восстанавливает реестр.

Другой сервис может узнать паспорт через GET /discover/{name} — получит описание, эндпоинты, схемы, но никогда адрес. Запрос отправляется через /route/{name}/{path} — MCGK проксирует, сохраняя заголовки, query string, тело, метод, стриминг. GET /map — все сервисы и их статусы. GET /observe — логи всех запросов. GET /health — результаты health-чеков.

Философия melting contract: вызывающий адаптируется к вызываемому, не наоборот. Вызывающий получает паспорт, читает схему, формирует запрос под неё. Если ответ не в ожидаемом формате — вызывающий адаптирует свой парсер. MCGK не трансформирует данные, не валидирует тела, не кэширует. Он только маршрутизирует и хранит паспорта.

Главный вывод из MCGK: полная изоляция адресов + паспорта с контрактами = система, в которой сервисы можно добавлять, убирать и заменять без каскадных изменений. Каждый сервис автономен, связь только через имена.

## Код без файлов: что мы поняли

Файловая система — это интерфейс для людей. Человек ориентируется по директориям, именам файлов, пространственной памяти ("auth где-то в src/utils"). LLM-агент не имеет пространственной памяти. Когда агент получает проект — он видит плоский поток текста. Файловая структура для него шум, а навигация по ней жрёт контекст, который мог бы тратиться на решения.

Когда агент работает с файлом на 500 строк, три четверти контекста уходит на то чтобы найти нужное место. Импорты занимают десятки строк и не несут логической информации — это метаданные сборки, а не метаданные смысла. Агент их читает, тратит токены, и не получает ничего полезного для решения задачи.

ida-procon уже решила эту проблему для мёртвого кода: агент не открывает файл, он делает GET /func-code и получает одну функцию. Меняет — POST /update-code. Метаданные хранятся отдельно в манифесте. Координатор собирает контур через /contour-code, склеивая функции в правильном порядке.

То же самое работает для живого кода. Код хранится в БД как граф нод. Нода — минимальная единица кода, содержащая хотя бы одно решение (ветвление, обработка ошибки, итерация с условием). Если код не содержит решения (чистая арифметика, форматирование, маппинг) — это micro, часть другой ноды, не самостоятельная единица. Давление сверху на размер ноды — тест (если нода не тестируема изолированно — слишком мелкая). Давление снизу — decision point (если нет ни одного решения — это не нода).

Каждая нода хранит чистый код без импортов, контракт (что принимает, что возвращает), список зависимостей (какие другие ноды вызывает), и спецификации четырёх уровней гранулярности. Уровни вдохновлены числами Данбара — вложенная структура социальных связей (5/15/50/150) где каждый уровень даёт меньше деталей. Для нод: as_is (полный код + все спеки), summary (полное описание без кода), ticket (15 строк — что делает, контракт, edge cases), name (одно предложение). Агент, работающий над нодой, получает её as_is, прямых соседей на уровне summary, дальних на уровне ticket, весь остальной граф на уровне name. Контекст дозируется по дистанции в графе.

Импорты генерируются автоматически при сборке. Скрипт обходит граф, резолвит зависимости, собирает файлы в формате целевого фреймворка, запускает бандлер. Файлы — промежуточный артефакт сборки, не источник правды. Как .o файлы при компиляции C.

Версионирование на уровне нод, не файлов. Агент меняет ноду — создаётся новая версия с task_id. Несколько агентов параллельно пишут разные версии одной ноды — турнир: каждая версия прогоняется через тесты с golden-окружением, лучшая побеждает. Не мерж текста, а отбор по поведению.

Один граф — много продуктов. Продукт — это подмножество нод плюс свои точки входа. Контур auth_flow используется в трёх продуктах. Добавить новый продукт — выбрать ноды, добавить недостающие, определить рёбра.

Проекция для человека генерируется по запросу: карта контуров, описания на человеческом языке, полный проект в файлах для скачивания. Человек не обязан читать код — он видит что система делает и почему, через спеки, не через исходники.

## Композиция: Contour Graph

Contour Graph — система, в которой MCGK является единым гейтом, а за ним стоят отдельные сервисы, каждый со своей ответственностью.

MCGK Gate — уже готов. Форк существующего MCGK. Все сервисы регистрируются через паспорта, общаются только по именам, адреса скрыты. Без изменений.

Claim-сервер — логика из ida-procon Claimer. Атомарные захваты нод с TTL, автоматический release при истечении, список захваченных нод. Агент не может работать с нодой, не claim'нув её. Два агента не могут claim'нуть одну ноду. Регистрируется в MCGK как "claimer".

Code-сервер — хранит ноды графа: id, код, язык, kind (contour/micro/config), контракт (accepts/returns), зависимости (imports), статус (draft/golden/deprecated), версии. Хранит рёбра между нодами (calls/uses/extends/tests). CRUD операции, поиск похожих нод для реиспользования. Регистрируется в MCGK как "codegraph".

Spec-сервер — хранит спецификации нод на четырёх уровнях гранулярности (name/ticket/summary/as_is). Формирует контекст для агента: по запросу собирает пакет из spec_as_is целевой ноды, spec_summary соседей, spec_ticket дальних, spec_name остальных. Регистрируется в MCGK как "specgraph".

Builder — получает граф из codegraph, для каждой ноды берёт golden-версию, генерирует импорты, собирает файлы, запускает бандлер. Три режима: build (весь продукт из golden), test_build (одна нода заменена на draft-версию, остальное golden), build_contour (standalone фрагмент для быстрой проверки). Регистрируется в MCGK как "builder".

Tester — получает собранный артефакт от builder, прогоняет тесты. Тесты — тоже ноды в графе с edge_type "tests". Регистрируется в MCGK как "tester".

Coordinator — верхний уровень. Назначает задачи агентам (/next-task — нода с наибольшим количеством непокрытых соседей). Перед созданием новой ноды проверяет через codegraph /similar есть ли подходящая. Формирует контекст через specgraph. Управляет жизненным циклом: draft → test_build → test → golden или reject. Управляет турнирами версий. Garbage collection: ноды без входящих рёбер и без golden-статуса удаляются по TTL. Регистрируется в MCGK как "coordinator".

Порядок поднятия: сначала MCGK, потом claimer, потом codegraph, потом specgraph, потом builder, потом tester, потом coordinator. Каждый следующий сервис зависит от предыдущих только через MCGK — по именам, не по адресам. Каждый можно поднять, протестировать и потрогать отдельно прежде чем двигаться дальше.
